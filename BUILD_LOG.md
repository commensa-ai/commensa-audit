## Session 7 notes (0.3.0) — extraction limiting

**Why:** the OSS-sweep protocol needs to audit large repos without walking their whole history, and an unbounded default is a footgun on a 10k-PR repo. Promoted from the ERPNext one-off to a pre-launch release.

**Flags (both optional, newest-first):** `--since YYYY-MM-DD` (inclusive UTC date on created_at) and `--max-prs N` (keep the N newest; **default 500**, `0` = no cap).

**Design — the cap lives at the CLI, not the library.** `GitHubExtractor.select_pull_numbers(since, max_prs)` still defaults to None/None = unlimited (the Gate A behaviour, byte-identical); the 500 default is an argparse default the CLI passes in. So `import`-level callers are unchanged and Gate A reproduces — only the CLI is bounded. When a limit is active the list call switches to `sort=created&direction=desc` and pagination early-stops on the cap or the first PR older than `since`, so a 10k-PR repo costs ~N list rows + detail calls, not its whole history. `self.capped` is set true only when more PRs existed than were returned; the CLI prints a raise-it notice then. Added `self.requests` (HTTP round-trip counter) for the "API calls used" line.

**Gate A (the locked regression):** the new default path (cap 500 + sort=desc; 162 < 500 ⇒ no truncation) was field-checked against `reference/orderwebv2_units.csv` — 162/162 rows, **0 locked-field drift** (created_at/merged/lines_added/lines_deleted/changed_files/looks_revert), only PR-51/72/75/76 titles differ (the documented S1 Unicode-vs-pilot rows). A naive byte-diff first flagged those 4 as "DRIFT"; the field-by-field comparison is the correct gate and it passes.

**Tests:** 16 in `tests/test_extract_limits.py` (offline fake `requests.Session`): default-unchanged + no-sort-params, newest-N + desc sort, inclusive since, since+max combine, early-stop pagination (cap and since), 0 = no cap, capped true/false/exact-size, parser default 500, `--max-prs 0`, negative rejection, since-format rejection. **55/55 total.**
